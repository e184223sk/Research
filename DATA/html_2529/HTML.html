AssemblyDefinitionは、参照元もリコンパイルされてしまう

C#UnityAssemblyDefinition

エディタリフレッシュの際のコンパイルだけで1分くらい待たされるプロジェクトを触っています。処理待ちという無駄な時間が増えるばかりではなく、脳内のキャッシュは切れてしまい、コンパイルが面倒くさいからと動作確認をおろそかにしてしまいがちになってしまいます。

ちょっとでもこの時間を短くしたいのですが、普段書いているゲームロジックコードを参照するエディタ拡張やライブラリ（！？）が蔓延っており、適当に参照を切っていくだけではあまり早くなりませんでした。

よくよく調べてみると、リコンパイルをスキップするアセンブリが少なくなっていました。

TL;DR;

変更したファイルのアセンブリをリコンパイル、およびリコンパイルしたアセンブリに依存するアセンブリをリコンパイルする。

Assembly-CSharpは自動で、すべてのアセンブリにリンクされます、依存しています。

無理矢理相互参照を排除する

基本的にはインターフェイスなどで宣言と実装を分けて実装を別アセンブリに分ける方法をとっていくことおもいます。しかし、リフレクションで名前でアクセスすることで、実質参照していないことになり相互参照を解除することができます！（もちろん名前が変わった瞬間例外発生！！）

Asm1.cs
public class Asm1 : MonoBehaviour
{
    void Start()
    {
        Assembly asm = Assembly.Load("Asm2");
        Type type = asm.GetType("Asm2", true);
        object instance = type.GetField("I").GetValue(null);
        object text = type.GetField("field").GetValue(instance);
        UnityEngine.Debug.Log(text);
    }
}
Asm2.cs
/// <summary>
/// 諸悪の根源シングルトン！
/// </summary>
public class Asm2 : MonoBehaviour
{
    public static Asm2 I;
    public string field = "Hoge";

    void Awake()
    {
        I = this;
    }
}

Asm1.csとAsm2.csが別のアセンブリになっていてもこれで値を取り出し、メソッドをひっぱたくことができます。保守性は最悪ですが、コンパイル時間が大幅に短縮できるのであれば背に腹は代えられないかもしれません。
